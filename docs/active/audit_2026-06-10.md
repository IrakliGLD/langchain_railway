# Backend Audit — Enai AI Analyst (`langchain_railway`)

**Date:** 2026-06-10
**Scope:** Backend at `D:\Enaiapp\langchain_railway` (FastAPI query pipeline). The chat
UI / auth front-end lives in `D:\export_enai` (Supabase edge functions + React); only the
`chat-with-enerbot` edge function was reviewed here as the security boundary that calls this
backend.
**Nature:** Read-only review. **No code was changed.** Findings are observations and
recommendations only.

This audit covers (1) security, (2) logic / correctness bugs, (3) improvement suggestions,
and (4) a critique of the Ideal Decision Tree in
[`query_pipeline_architecture.md`](query_pipeline_architecture.md).

Severity legend: **High** = exploitable / data-correctness in production, **Medium** =
realistic risk or notable defect, **Low** = hardening / edge case.

---

## 1. Security Findings

### S1 — Multi-statement (stacked) SQL is not rejected — **Medium** (defense-in-depth)
`core/sql_generator.py::sanitize_sql` and `simple_table_whitelist_check` both validate via
`sqlglot.parse_one(...)`, which parses and returns **only the first statement**. The
*original* SQL string (not a re-rendered AST) is what's later passed to
`execute_sql_safely` → `conn.execute(text(sql))`. A string such as
`SELECT * FROM price_with_usd; SELECT * FROM auth.users` would pass validation (only the
first `SELECT` is inspected for tables/forbidden nodes) while the full string reaches the
driver.

- Writes are blocked by `SET TRANSACTION READ ONLY`, so `DROP`/`DELETE`/`UPDATE` in a second
  statement fail.
- A second **`SELECT` against a non-whitelisted table/schema** (e.g. `auth.users`) is **not**
  caught by the whitelist, because the whitelist only saw the first statement.
- Exploitation requires the LLM to emit stacked SQL (the user submits natural language, not
  SQL), so this is defense-in-depth rather than a direct injection. The firewall + prompt
  design make it unlikely but not impossible.

Recommendation: reject input that parses to more than one statement
(`len(sqlglot.parse(sql, read="postgres")) > 1` → 400), and/or execute the **re-rendered**
single-statement AST rather than the raw string. The strongest fix is **S7** below.

### S2 — Firewall is English-only; Georgian/Russian bypass it — **Medium**
`guardrails/firewall.py` block rules (`instruction_override`, `prompt_exfiltration`,
`role_hijack`) are English regexes (`ignore … previous … instructions`, etc.). The product
explicitly supports Georgian and Russian (`utils/language.py`, multilingual conceptual
patterns throughout). A jailbreak / prompt-exfiltration attempt phrased in Georgian or
Russian matches none of the block rules and is passed straight through to the analyzer/LLM.
The downstream grounding/provenance gate only catches numeric hallucination in *narrative*
answers — it does not catch instruction-override.

Recommendation: add Georgian/Russian variants of the block patterns, or replace the regex
firewall with a small model-based intent classifier. At minimum, document the gap.

### S3 — In-memory rate limits + sessions are per-process — **Medium**
`main.py` rate-limit buckets (`_preauth_/_gateway_/_user_rate_buckets`) and
`utils/session_memory._SESSION_STORE` are plain in-process dicts. Under more than one uvicorn
worker or more than one Railway replica:
- Effective rate limits are multiplied by the worker/replica count (a `10/min` public limit
  becomes `10 × N`).
- Sessions issued on one worker are unknown to another, so server-bound conversation history
  becomes inconsistent (the edge function partly compensates by re-seeding from
  `chat_history`, but the signed session token then resolves to an empty store on the other
  worker).

Recommendation: move both to a shared store (Redis) for multi-instance correctness, or
explicitly pin the service to a single worker and document that constraint.

### S4 — Rate-limit bucket keys accumulate without eviction — **Low/Medium**
`_check_sliding_window_rate_limit` trims *timestamps within* a key but never removes empty /
stale keys. Over time the three bucket dicts grow one entry per distinct IP / session seen
(unbounded). Session store has TTL cleanup; rate buckets do not.

Recommendation: periodically prune keys whose timestamp list is empty after trimming.

### S5 — Pre-auth limiter keys on `get_remote_address`, not XFF — **Low**
`_check_preauth_rate_limit` keys on `get_remote_address(request)` (the socket peer). Behind a
proxy/CDN (Railway typically terminates in front), the peer is the proxy, so **all**
unauthenticated traffic shares one bucket → the 300/min preauth guard is effectively global
and can be exhausted by one noisy client, or conversely never reflects per-client abuse. The
*gateway* limiter does read `x-forwarded-for`, which is the opposite trust assumption.

Recommendation: standardize on a trusted forwarded-IP extraction (with a configured trusted
proxy hop count) for the preauth limiter too.

### S6 — Caller-supplied `conversation_history` trusted in public-bearer mode — **Low/Medium**
When `ENABLE_PUBLIC_BEARER_AUTH` is on, a holder of a valid Supabase JWT can call `/ask`
directly (bypassing the edge function, which is the component that loads history from the DB
and drops client-supplied history). `main.py` then seeds `bound_history` from
`q.conversation_history` when the in-process session is empty. This is a prompt-injection
surface into the analyzer/summarizer context. It is bounded (≤3 turns, ≤2000 chars each), and
the firewall does **not** inspect history content.

Recommendation: in bearer mode, either ignore client history entirely (mirror the edge
function) or run history items through the firewall/length caps it already applies to `query`.

### S7 — DB connection privilege relies on app-layer read-only enforcement — **Medium (strategic)**
`SUPABASE_DB_URL` appears to be a broadly-privileged connection; the only thing preventing
writes/secondary reads is `SET TRANSACTION READ ONLY` plus the app-layer table whitelist. If
S1 (or any future validation gap) is bypassed, the database itself imposes no boundary.

Recommendation: connect as a **least-privilege Postgres role** that has `SELECT` only on the
nine whitelisted views/matviews and nothing else. Then the whitelist and read-only
transaction become true defense-in-depth instead of the sole control. This single change
neutralizes most of S1.

### S8 — Error detail leakage in gateway mode — **Low**
On pipeline failure, public-bearer callers get a generic `"Query processing failed"`, but
gateway callers get `detail=f'Query processing failed: {e}'` (raw exception text, which can
include SQL fragments / schema names). The gateway is internal, but the text also lands in
edge-function logs and any client of the gateway.

Recommendation: log the exception server-side; return a generic detail to all callers.

### S9 — Referer check provides little protection — **Low**
The `Referer` check in `ask_post` only runs *if* a Referer header is present; omitting it
skips the check. CSRF protection for a token/secret-authenticated JSON API is largely
redundant anyway. Not harmful, but it is security theater as written and can cause false
403s for legitimate clients that send a Referer from a non-listed origin.

### Positives
- Gateway secret and admin secrets use `hmac.compare_digest` (constant-time).
- Session tokens are HMAC-signed; caller-provided session IDs are not trusted without a valid
  signature.
- Typed tools (`agent/tools/*`) build SQL with **bind parameters** and validate
  metric/currency/entity/granularity against hard-coded enum allowlists — no string
  interpolation of user values. This is the safest part of the SQL surface.
- `/metrics` and `/evaluate` are env-gated, admin-secret-gated, and `/evaluate` additionally
  refuses to run outside `development`/`test`.
- Backpressure gate + circuit breakers are correctly try/finally-scoped (no double-release on
  the early-return auth/rate-limit paths).
- No `eval`/`exec`/`os.system`/`subprocess`/`pickle` in production code.

## 2. Logic / Correctness Findings

### L1 — Supply-side filter silently drops rows on the legacy SQL path — **Medium**
`agent/sql_executor.py::validate_and_execute` post-filters any result containing `type_tech`:
if the query mentions demand/loss/export it keeps DEMAND rows, if "transit" it keeps TRANSIT,
**otherwise it defaults to SUPPLY only**. A question about *total* electricity (supply + demand
combined) silently has demand rows removed, producing an understated total with no warning to
the user or the summarizer. The branch only skips filtering when the filtered subset is empty.

Recommendation: only apply the side filter when the query clearly targets one side; for
total/ambiguous queries, keep all rows (or surface that a side was selected).

### L2 — `coerce_to_psycopg_url` replaces scheme substring anywhere — **Low**
`core/query_executor.py`: `url.replace(parsed.scheme, "postgresql+psycopg", 1)` replaces the
first occurrence of the scheme *string* in the whole URL, not specifically the `scheme://`
prefix. If a username/password/host contained `postgres`/`postgresql` as a leading substring,
the URL could be corrupted. Edge case, but `urlunparse`/explicit prefix handling is safer.

### L3 — `LIMIT` enforcement misses outer query when a subquery has LIMIT — **Low**
`plan_validate_repair` appends `LIMIT MAX_ROWS` only when `LIMIT_PATTERN` finds no `LIMIT` in
the string. A query whose `LIMIT` is in an inner subquery (but not the outer SELECT) is
treated as "already limited," leaving the outer result uncapped. Mitigated by the
`fetchmany`/`MAX_ROWS` truncation in `execute_sql_safely`, so impact is bounded — but the
two mechanisms can disagree on row count.

### L4 — Synonym regex rewrite runs on raw SQL after validation — **Low**
`plan_validate_repair` applies `SYNONYM_PATTERNS` (`\bprices\b → price_with_usd`, etc.) by
regex over the whole SQL string, including inside string literals or column aliases. It runs
*after* whitelist validation, so a rewrite is not re-validated. Targets are all whitelisted so
it can't reach an unauthorized table, but it can corrupt a query where those words appear as
data values.

### L5 — `WINTER_MONTHS` includes Aug–Oct — verify domain intent — **Low (verify)**
`config.py`: `SUMMER_MONTHS = [4,5,6,7]`, `WINTER_MONTHS = [1,2,3,8,9,10,11,12]`. Classifying
August, September, and October as "winter" is unusual; it may be a deliberate hydrology/season
definition for the Georgian market, but it is surprising enough to warrant an inline comment
or confirmation, since seasonal averages depend on it.

### L6 — `_DropUvicornAccess404` depends on positional log args — **Low**
`main.py`: `int(record.args[4]) != 404` assumes uvicorn's access-log arg layout. A uvicorn
version change silently breaks the filter (it's caught and returns `True`, so noisy but not
fatal). Fragile coupling.

### L7 — Conceptual / data-intent detection is brittle multilingual regex — **Low**
`utils/query_validation.is_conceptual_question` and the topic extractors are large hand-tuned
regex/keyword tables across three languages. With the analyzer authoritative this is mostly a
fallback, but `validate_sql_relevance` / `should_skip_sql_execution` still consult it on the
legacy path, and small wording changes flip routing. This is the same brittleness the
architecture doc says it moved away from — it still lives on the fallback path.

## 3. Improvement Suggestions

1. **Least-privilege DB role (S7)** — highest-leverage single change; makes the whole SQL
   safety stack defense-in-depth.
2. **Reject multi-statement SQL (S1)** — cheap, closes a real gap.
3. **Externalize sessions + rate limits to Redis (S3)** — required before horizontal scaling;
   today the limits and session memory are silently per-process.
4. **Multilingual guardrails (S2)** — the firewall must cover the languages the product
   accepts.
5. **Grounding check for deterministic renders** — see A6 below; today the provenance gate is
   a no-op for the generic renderer, so the deterministic shapes (SCALAR/LIST/TIMESERIES/
   COMPARISON/SCENARIO) ship with no post-hoc value-vs-evidence check at all.
6. **Remove or re-document orphan code** — `generic_renderer._render_forecast` is described in
   the architecture doc as orphaned from the production path; either delete it or mark it
   clearly, so the "ideal tree" doesn't document a dead branch as if live.
7. **Pipeline failure should degrade, not 500** — the SQL-validation path already degrades to a
   conceptual answer (good); consider the same graceful-degradation discipline for analyzer
   and summarizer failures so a single bad LLM response doesn't return a 500.
8. **Prune stale rate-limit keys (S4)** and **standardize trusted-proxy IP extraction (S5)**.
9. **Pin magic thresholds to evidence** — `ANALYZER_CONFIDENCE_OVERRIDE_THRESHOLD=0.8`, the
   legal-list `0.85`, `TOOL_RELEVANCE_OVERLAP_THRESHOLD=0.3`, etc. are env-tunable but have no
   feedback loop; consider logging the cross-check disagreement rate so these are tuned from
   data rather than intuition.

## 4. Critique of the Ideal Decision Tree (`query_pipeline_architecture.md` §2.3)

The document is unusually disciplined — it names the code as source of truth and tracks its
own debt. The following are where the *Ideal Tree* over-claims or is internally inconsistent.

### A1 — "Stage 0.2 is the one LLM call" is not true, and the doc itself contradicts it
The stated **design principle** is "Stage 0.2 is the one LLM call. Everything after just
executes." But the same document describes: the Stage 4 **narrative LLM summarizer**, the
**FORECAST path that always goes through the LLM** (since 2026-05-22), an **OpenAI fallback on
Gemini failure**, and re-ranked vector retrieval. There are several LLM calls per request. The
honest principle is "**one interpretation call** — the question is interpreted once and emits a
contract." The "one LLM call" phrasing oversells determinism and will mislead a new reader
about latency/cost and failure modes.

### A2 — The "single execution loop (TARGET)" is internally contradictory
§2.3 says the merge of primary execution and the Stage 0.8 secondary loop "into one for-loop
is open," and §3.6 describes `_execute_evidence_plan` as **three logical passes** (primary
strategy chain + secondary loop + driver enrichment). Yet §5.1 and §6 declare the single-loop
target "no longer aspirational — it is the structure that function implements." A
three-pass function wrapped in one outer function is not a single loop. Either the target
should be restated as "one orchestration *function*" (true) or §6's claim should be walked
back. As written the two sections disagree.

### A3 — Four-strategy primary picker contradicts "deterministic via candidate_tools[0]"
§2.3 asserts tool selection is "**deterministic** via `candidate_tools[0]` + evidence planner,"
but §3.6 documents `_pick_primary_invocation` as a **four-strategy fallback chain**
(plan-driven → keyword router → analyzer-built → authoritative router fallback). That is a
priority cascade with legacy keyword routing inside it, not a deterministic single source. The
tree presents more determinism than the code has.

### A4 — Multiple live legacy fallback layers under the "contract executes" story
Beneath the contract there are still: the **agent loop** (`run_agent_loop`), the **legacy SQL
escape hatch** (Stages 1/2), and **source-aware failure recovery**
(`_attempt_analyzer_tool_recovery`). The tree frames these as edge fallbacks, but they are
real, reachable code paths with their own routing logic. Stage 0.7 removal (§5.2) is gated on
"two weeks of production data" with **no owner and no date** — an open-ended gate is, in
practice, permanent. The tree should either schedule that decision or stop describing 0.7 as
transitional.

### A5 — The central bet (trust the contract) has thin runtime correctness checking
The design trades regex brittleness for LLM non-determinism (§5.3 concedes this is a "standing
quality area"). The only runtime guards on a *wrong* contract are: `_cross_check_answer_kind`
(a heuristic comparing the LLM's `answer_kind` to a `query_type`-derived one — i.e. another
heuristic), the evidence validator (shape check), and the provenance gate (numeric, narrative
only). None of these catch "right shape, wrong tool/metric" — a confidently-rendered wrong
answer. The cross-check, legal-list exception, and the scenario-metric "quantitative anchor"
gate are exactly the kind of accreting special-case logic the design claims to have escaped.
The tree should acknowledge that misclassification has **no deterministic backstop**, only
mitigations.

### A6 — The provenance gate is narrative-only; deterministic renders are unguarded
§3.9 / §5.5 note the provenance gate returns `no_claims` (pass) for canonical-frame paths. So
every deterministic shape (SCALAR/LIST/TIMESERIES/COMPARISON/SCENARIO) — the majority of "happy
path" answers — has **no post-hoc grounding check**. If an evidence frame is built from the
wrong rows, the renderer prints the wrong number with full confidence and the safety net does
not fire. The tree presents the provenance gate as the grounding backstop without making clear
it covers only the narrative minority of answers.

### A7 — Vector-retrieval tiering compounds analyzer errors
§3.3: `render_style=DETERMINISTIC` (data) → **SKIP** vector knowledge. If the analyzer
mis-labels a question that genuinely needs regulatory/conceptual grounding as deterministic
data, knowledge retrieval is skipped entirely, so the answer is silently ungrounded. Tiering
is a good cost optimization, but it makes a Stage 0.2 misclassification *more* damaging, not
less — the tree should note this coupling as a risk, not just an efficiency win.

### A8 — Documentation drift risk is structural
The doc is dated 2026-05-13 and several "DONE / no action required" sections (§5.4, §5.5) are
rationales rather than verifiable assertions. Because the doc explicitly defers to code as
source of truth, every such prose rationale is a future drift point. Recommend reducing §5's
"no action required" essays to one-line pointers to the governing test, so the doc stays
falsifiable.

### What the tree gets right
- Emitting the full answer contract once and pushing interpretation upstream is sound and does
  reduce per-stage re-guessing.
- The evidence-frame + generic-renderer split genuinely removes per-question Stage 4 patches
  for the standard shapes.
- The doc is honest about its open debt (§5) and pins invariants to tests — better than most.
